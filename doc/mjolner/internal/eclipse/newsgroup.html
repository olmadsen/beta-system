Dave Thomson <Dave_Thomson@sympatico.ca>:
It depends on what kind of code you want to integrate.
If you have a bunch of native libraries that are non-UI, then
it should be fairly straightforward to integrate with JNI.
SWT also provides COM support that you can use to integrate
via COM on windows if you want to make COM controls, but the
JNI approach is more straightforward.
For UI code, on Windows you can either implement new SWT widgets
to wrapper native window handles or if you make an ActiveX
control out of your native code, SWT provides ActiveX
integration capability.
For other platforms, you can write a new SWT widget to wrapper
native window handles much the same way SWT wrappers native
widgets on those platforms.
If you have widgets that you have already implemented natively,
and want to reuse them, then writing a new SWT widget to wrapper
these native widgets may be the best approach.
There is an article on eclipse.org that talks about building
new SWT widgets like this.  The article is called:
"Creating Your Own Widgets using SWT"
There is also an article on the ActiveX support:
"ActiveX Support in SWT"
I didn't post the links because they are long and gets wrapped.
Just click on articles in the nav bar and scroll down to the SWT
section.
Hope this helps.

 

Erich Gamma <erich_gamma@oti.com>: 
Integrating command line versions of existing tools is
another possible path. Obviously this means you
will loose the existing IDE functions of your tool set, but
this could be a quick start to enable BETA development
under Eclipse.
We went down this path for a basic C plugin
that is part of the IBM VisualAge Micro Edition.
This required implementing the following Eclipse components:
* building/compiling: implement an Eclipse Builder that calls a "make"
command.
   Analyze the output of make and generate problem markers.
* editing: implement a C editor and a parser to populate and
    reconcile the outline view.
* searching: only text searching is supported, you get it for free
   from Eclipse.
* running: implement a launcher that can run an executable.
* debugging: the platform provides a language independent debugging model.
   Once you implement this model API you get the debugger UI almost for
   free. We didn't get to this for the C plugin, mostly because of the
   lack of published APIs to existing C debuggers.
