Directory structure for BETA Projects

This is a short set of rules for the organization of each directory in $BETALIB.

The existing directory objectserver will be used as an example.

Structure

The structure of $BETALIB/objectserver should be (objectserver is only used as an example - it does not look exactly like this on disk): 
  $BETALIB/objectserver
     |
     |___ CHANGES
     |___ TODO
     |___ *.bet
     |___ private
            |___ *.bet
            |___ external
                   |___ *.c, *.h, *.make
     |___ demo
     |___ doc
     |___ test
     |___ ...
The following sections explain the different files/directories.

CHANGES

CHANGES should contain a description of all changes to the library/tool, that affects the user. That is, at least interface changes and semantic changes.

The CHANGES file follows the versions (and may be under CVS). It is optional whether the CHANGES file should describe changes for all previous versions (i.e. be cummulative) or just for the version it belongs to.

TODO (optional)

Contains informal description of known bugs and planned changes for the tool/library.

private

The private directory contains implementation files (BODY-files) for the interface files that reside in the directory above private.

There should be no files in the private directory, that are not referenced by the interface files (this will make people INCLUDE files from the private directory - a very bad practice!).

The private directory may contain subdirectories, if the implementation is best split into parts this way.

external (optional)

The external directory contains external files such as C or assembler files, and the Makefiles needed to keep them up-to-date.

demo

The demo directory should contain small self-explaining, well-commented demo programs, that use the library/tool.

The demo programs should contain complete paths (relative to $BETALIB), so that the user can copy it to his/her own directory, and immediately compile the program.

If appropriate, the demo directory may contain subdirectories for categorizing the demos.

Notice, that test programs should not reside in the demo directory - they belong in the test directory.

doc (optional)

This directory may contain internal as well as external documentation about the tool/library.

E.g. there may be text files describing the implementation details, and html- or text files used by the tool/library for online-documentation.

test (optional)

The test directory contains files used by the implementor to test the tool/library. Often these files are only meaningful to the implementor, and one should avoid the temptation to move them to the demo directory, unless they are properly rewritten and documented.
