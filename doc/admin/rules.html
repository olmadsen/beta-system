Rules for editing miadoc pages

Since a lot of people are involved in maintaining the document pages, a few basic rules must be agreed on:
  • Edit your own copy
    Do not edit in any of the places this documentation is officially checked out.
    Instead get your own copy by
      cvs get doc
    This rule is to prevent simultaneously editing and to prevent external users from experiencing inconsistent files.

  • Using a Web editor
    If you prefer using a high-level Web editor instead of editing HTML directly, this is OK.
    However, you should only use such an editor for pages you are the main editor of, and especially not for pages that are potentionally edited by several people (like the main index.html, the document-list.html, etc.).
    The problem with using a Web editor with pages that can be edited by other people is, that the Web editors tend to do a complete rewite of the HTML mark-up (e.g som uses uppercase tags, others lowercase; some eidtors clutter the HTML with &nbsp; etc.).
    This gives raise to huge virtually unsolvable CVS conflicts.

  • Use relative paths in URLs
    Use releative paths in all links to other parts of this documentation. That is use, e.g.,
      <IMG SRC="../images/warning.gif>
    instead of
      <IMG SRC="http://www.daimi.au.dk/~beta/doc/images/warning.gif>
    This is to make doc as self-contained as possible, since it will at some time be packed onto a CD from which a user should be able to browse the pages locally.

  • Add GIF files as binary
    This is more a hint than a rule: GIF files and other binary files must be added as such to CVS:
      cvs add -kb foo.gif
    See CVS: Handling Binary Files for further details.

  • Build Interface Descriptions on UNIX
    Again just a hint: The interface files are build on UNIX by issuing
      make
    from the main directory of this documentation. This build involves Bourne Shell scripts, which will only work on UNIX.
